# 版本更新与回滚规则

## 核心原则

项目规则发生大改时，系统应优先生成新的本地备份，不默认覆盖旧文件，也不默认上传 GitHub。

## 本地备份规则

每次大改规则时：

1. 生成新的 zip 备份。
2. 不默认覆盖旧 zip。
3. 文件名建议包含日期或版本号。

推荐命名：

```text
prompt-script-master-backup-YYYYMMDD.zip
prompt-script-master-v版本号.zip
```

## 状态记录规则

每次大改规则后，应更新 `PROJECT_STATUS.md`。

记录内容包括：

- 修改日期
- 修改模块
- 修改原因
- 是否影响旧规则
- 是否需要同步 Skill
- 当前 GitHub 状态

## GitHub 上传规则

上传 GitHub 前必须询问用户。

不能因为生成了 zip 就自动上传。

必须问：

```text
这次修改你希望只在本地项目中使用，还是上传到 GitHub？
A. 只在本地使用
B. 更新本地 zip 备份
C. 上传到 GitHub
D. 本地更新后，等我确认再上传 GitHub
```

## 覆盖旧项目规则

如果用户选择上传到 GitHub，系统不能直接覆盖原项目。

必须继续询问：

```text
你希望覆盖原 GitHub 项目，还是作为新版本 / 新分支 / 新仓库上传？
A. 覆盖原项目
B. 新建一个版本分支
C. 新建一个 release 版本
D. 新建一个独立仓库
E. 只生成 zip，不上传
```

## 安全底线

1. 没有明确同意，不上传 GitHub。
2. 没有明确选择覆盖，不覆盖旧项目。
3. 没有明确要求覆盖本地文件，不覆盖旧 zip。
4. 规则大改后优先生成新备份。

## 回滚规则

如果用户发现新规则不好用，可以回到之前备份。

系统应保留旧 zip 的下载路径或版本记录，并允许用户说：

- 回到上一个版本
- 恢复 20260621 备份
- 不要这次修改
- 重新基于旧版本修改

## 最终标准

本地保存可自动执行。  
GitHub 上传必须确认。  
旧项目覆盖必须二次确认。  
版本可追踪，可回滚。
